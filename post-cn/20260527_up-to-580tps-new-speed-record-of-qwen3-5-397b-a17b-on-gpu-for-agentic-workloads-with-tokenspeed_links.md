## TL;DR

TokenSpeed 推理引擎在 GPU 上运行 Qwen3.5-397B-A17B 模型，实现了突破性的 580 tps 记录。这一针对 Agentic 工作负载的极限性能，源于系统性消除内存拷贝、先进的内核融合，以及完全重叠的 CPU-GPU 执行——始终保持 GPU 满负荷运转。在功能层面，TokenSpeed 还支持混合前缀缓存和统一的 Prefill-Decode 状态传输，以应对复杂的 Agentic 服务场景。

## 2. 运行时设计与功能

Qwen3.5 采用混合架构：大多数层为 GDN（具有每层 conv_state 和 temporal_state 的线性注意力），每隔 N 层为标准全注意力层，配备传统 KV 缓存。TokenSpeed 在前缀缓存、调度和 Prefill-Decode 分离方面提供完整的 GDN 感知支持，实现对整个混合栈的高效服务。

### 2.1 GDN/Mamba 前缀缓存

前缀缓存对 Agentic 工作负载至关重要，其中多轮工具调用序列频繁共享长上下文和对话历史。TokenSpeed 的前缀缓存分为两层。C++ 负责逻辑缓存：基数树匹配、页面 ID、驱逐和 Mamba 槽位生命周期。Python 负责物理张量：GPU KV 页面、Mamba `conv_state` / `ssm_state`、流排序、写时复制、归零和快照复制。

对于普通 KV 缓存，前缀命中意味着复用缓存的页面 ID。但对于 Mamba 来说，这还不够。可复用的前缀还必须在相同前缀边界携带循环状态。TokenSpeed 通过将 `MambaSlot` 附加到代表缓存 KV 前缀的同一基数树节点来解决这个问题。

#### 槽位生命周期

每个活跃的 Mamba 请求可持有两种槽位类型：

- `working` 槽位：当前前向步骤使用的可变状态。
- `checkpoint` 槽位：快照目标，之后可发布到前缀树。

调度器在 C++ 中分配这些槽位，但 Python 负责写入实际张量内容。

一个 checkpoint 槽位只有在两件事都为真时才变为可复用：Python 已用干净状态填充它，且 C++ 已将其附加到块对齐的基数树节点。

#### 前缀匹配与写时复制

当未来的请求匹配树时，`HybridPrefixCache` 首先执行正常的 KV 前缀匹配，然后找到最近的 Mamba checkpoint 节点。如果存在这样的节点，调度器返回 `mamba_cow_src_index`。

Python 随后在运行前向之前，将该缓存 checkpoint 复制到请求的私有工作槽位中。缓存的树槽位不被修改，只有请求的工作槽位会改变。

#### 保持 Checkpoint 干净

主要的正确性风险是复用槽位中的过时数据：`MambaChunkAllocator` 分发整数槽位 ID 时不清除 GPU 内存。TokenSpeed 通过运行时规则防止过时状态。

具体而言，新分配的工作槽位保证安全的方式恰好有两种：它要么从已知干净的 checkpoint 接收写时复制副本，要么 Python 在使用前显式将其归零。Checkpoint 仅在对齐边界发布，因此树永远不会将任意中间状态作为可复用前缀状态进行广告。

#### 重叠调度下的分块 Prefill

分块 Prefill 在重叠模式下引入了一个微妙之处：CPU 可能在提交前一块输出之前就调度下一块。由于 CUDA 流排序，checkpoint 仍然是安全的。

前一块的 Mamba 前向在 `execution_stream` 上写入 checkpoint。在下一次循环迭代开始时，默认流等待 `execution_stream`。只有在那之后，C++ 才将前一块插入树中并分离其 checkpoint 槽位。下一块随后获得一个新的 checkpoint 槽位。

关键不变式是：
_C++ 可能在重叠调度期间发布 checkpoint 槽位 ID，但任何后续 GPU 消费者都在前一块的快照写入之后排序，且已发布的槽位不再作为下一个 checkpoint 目标被复用。_

#### Decode 重叠

Decode 存在不同的危险：在 CPU 提交前一结果之前，下一次 decode 可能修改同一工作槽位。TokenSpeed 通过在分派下一次 decode 之前对块对齐的 decode 状态进行快照来处理这个问题。

这在工作槽位推进之前保留了干净的状态。

#### 总结

TokenSpeed 的 Mamba 前缀缓存是安全的，因为它将 Mamba 状态视为树所有的 checkpoint，而非请求的附带副作用。C++ 控制槽位何时成为前缀树的一部分。Python 控制张量内容何时被复制、归零和快照。两者共同维持一个核心不变式：从前缀树可达的每个 Mamba 槽位都包含该树节点所代表前缀的干净、对齐状态。

### 2.2 调度器

混合架构对调度器提出了独特要求：它必须同时管理 KV 缓存（全注意力层）和 Mamba 状态（线性注意力层）作为两个独立的资源池。

#### Mamba 状态与混合模型管理

TokenSpeed 的调度器实现了以下关键机制：

- **双资源池管理：** 每个请求同时持有 KV 缓存块索引和 Mamba 池槽位索引（`mamba_pool_indices`），调度器负责管理两者的分配和释放。
- **状态生命周期：**
  - 请求到达时：分配 mamba_pool 槽位
  - Prefill 期间：填充初始状态（或从前缀缓存加载）
  - Decode 期间：每步原地更新状态
  - 完成或被抢占时：释放槽位
- **投机解码支持：** 调度器维护中间状态缓存（`spec_cache`），存储每个投机步骤的 Conv/SSM 状态快照，在验证失败时支持回滚。
- **层级路由：** `HybridLinearAttnBackend` 基于 `layer_id` 将前向调用路由到适当的后端（全注意力或线性注意力），并为每种后端类型分别初始化元数据。

### 2.3 GDN PD

#### 2.3.1 挑战

对于混合模型，Mamba 层除了维护传统键值对之外，还维护状态张量。这些状态必须与 KV 缓存一起从 Prefill 节点传输到 Decode 节点，要求全注意力层和 Mamba 层之间正确的逐层对齐。

#### 2.3.2 我们构建的内容

我们为 PD 分离引入了端到端的 Mamba 缓存支持，包括：

**1. 统一状态传输：两个世界，一条线路**

核心洞见是，Mamba 状态尽管语义不同，但可以使用与 KV 缓存相同的 RDMA 机制进行传输——只要系统知道如何寻址它们。

我们在每个节点上设计了双张量池：一个池持有卷积状态（因果卷积的短期记忆），另一个持有循环 SSM 状态（长期压缩历史）。两者都预分配为连续的 GPU 内存，每个请求在每层拥有恰好一个槽位。在注册时，Prefill 和 Decode 节点交换缓冲区描述符——基地址、每槽大小，以及从每个物理缓冲区到其对应全局层 ID 的映射。

传输开始时，系统将每个请求的槽位索引映射到物理字节偏移，将连续槽位分组为分散-聚集块，并以批量 RDMA 写入方式发出。从网络的角度来看，Mamba 状态只是另一组内存区域——无需序列化，无需中间暂存。关键区别在于寻址方式：KV 缓存通过页表索引，而 Mamba 状态通过调度器分配的平面槽位 ID 索引。

**2. 跨层调度：统一心跳**

难题中最微妙的部分是**何时**传输每层的状态。

在逐层传输模式下，Prefill 节点不等待整个前向传递完成再开始数据移动。相反，它在每个层组完成后立即开始传输数据——将计算与通信重叠。但对于混合模型，这意味着传输线程必须像跟踪一条连续管道一样，跨注意力层和 Mamba 层追踪进度。

我们引入了一个**统一步骤计数器**，每层前向传递后递增一次——无论类型。传输线程监视此计数器，并对每个层窗口发送属于该窗口的数据：全注意力层的 KV 页面，Mamba 层的状态槽位。模型的层类型模式对传输逻辑不可见——它只是询问"哪些缓冲区映射到第 4 至 7 层？"，并在计数器达到 7 后发送所有数据。

在 Decode 端，此机制的镜像是**层完成屏障**：模型前向可以在第 15 层状态到达之前开始执行第 0 层。每层的计算调用状态池，仅当该特定层尚未加载时才阻塞。这允许 Decode 将网络接收与早期层执行重叠，将传输延迟隐藏在有用工作后面。

**3. PD 感知的 Token 生命周期：三阶段握手**

最后一部分将状态传输与 Token 生成生命周期连接起来。在分散式系统中，Prefill 节点不仅产生状态——它还产生第一个输出 Token。Decode 节点在开始生成之前需要两者。

我们设计了三阶段握手：

1. **传输完成：** 最后层组的所有 KV 页面和 Mamba 状态已传输。但传输线程尚未宣告成功——它在屏障处等待，等待前向传递完成。
2. **Token 产生：** Prefill 前向完成并发出第一个输出 Token。事件循环记录此 Token 并通知等待的传输线程。
3. **状态交付：** 传输线程通过侧信道向 Decode 端点发送轻量级状态消息（携带引导 Token）。只有当 Decode 节点同时收到批量状态数据和此 Token 时，它才向其调度器发出"远程 Prefill 完成"事件。

此协议确保一个不变式：**Decode 节点永远不会以不完整状态开始生成，也永远不会浪费一步重新推导第一个 Token**。Mamba 状态、KV 缓存和引导 Token 作为逻辑原子单元到达——即使它们通过不同路径、在不同时间传输。

## 3. 性能优化

### 3.1 Mamba 状态更新优化

#### 通过索引间接消除 Mamba 状态复制

在基于 Mamba 式线性注意力的投机解码中，目标-验证阶段传统上带有隐藏的内存成本。在草稿模型产生投机 Token 后，基础模型向前运行以验证它们。由于每个草稿 Token 将 Mamba 状态推进一步，引擎需要为每个投机位置保留中间状态，然后根据接受了多少 Token 来恢复正确的状态。

之前的管道使用专用的中间状态缓存来处理：在验证期间，内核将每步 Mamba 状态写入侧缓冲区，验证后的 `fused_mamba_state_scatter_with_mask` 内核将接受位置的状态复制回调度器拥有的工作槽位。散射本身是跨 `num_layers × state_dim` 的完整张量复制——这并非免费，且在每个解码步骤都执行。

#### 核心思想：移动指针，而非数据

我们不再将中间状态缓冲到单独的缓存中再散射接受的状态，而是让内核将每步输出直接写入专用的物理行，然后简单地记住_哪一行_持有规范状态。

状态缓冲区扩展了一个**草稿区域**，附加在调度器分配的基础槽位之后；每个请求拥有一个由其 `req_pool_index` 索引的私有草稿行切片。轻量级表 `current_input_indices` 记录每个请求当前哪个物理行持有其规范 Mamba 状态。

在目标-验证期间：

- **输入重定向：** 内核从 `current_input_indices` 中记录的行读取其初始状态（可能是工作槽位、COW 分叉槽位或之前步骤的草稿行）。这里不发生数据移动——只有索引查找。
- **输出路由：** 每请求的 `output_state_indices` 张量精确告知内核每步输出的写入位置：槽位 0 是工作行，槽位 1..N 是请求私有的草稿行。内核直接写入这些预分配的位置，完全消除了中间缓存。
- **验证后记账：** 一旦接受长度已知，我们只需将 `current_input_indices[req]` 更新为指向对应最后一个接受 Token 的草稿行。这是一个 **O(1) 整数写入**，而非 O(L·D) 张量复制。

### 3.2 运行时优化

#### 3.2.1 重叠即一切

遵循现代推理引擎的通用实践，TokenSpeed 采用 CUDA 多流并行来重叠非顺序操作。通过在多个流中并发执行独立工作负载，TokenSpeed 有效减少调度开销并提升端到端延迟。

**共享专家与路由专家重叠**

Qwen3.5 MoE 层包含共享专家和路由专家。共享专家处理所有 Token，而路由专家仅处理 TopK 选择的 Token。两者天然可并行化，通过 `StreamFork` 类实现流分叉和同步：

1. 主流执行 TopK 路由、专家分派和 MoE GEMM
2. 辅助流并发执行共享专家前向（gate_up → SiLU → down）和 sigmoid 门控
3. 两个流在合并结果前通过事件同步

这种重叠隐藏了共享专家的计算延迟，降低了生产部署中单个 MoE 层的时间。

**GDN 输入投影双流优化**

GatedDeltaNet 层的输入投影包含两个独立的线性层（`in_proj_qkvz` 和 `in_proj_ba`），同样在流之间并行执行。

此优化仅在 CUDA 图捕获期间激活，其中较小的 `in_proj_ba` 投影在备用流上完全隐藏在较大的 `in_proj_qkvz` 之后。

#### 3.2.2 融合越多，延迟越低

**Gemma AllReduce 融合**

GemmaRMSNorm 使用 `x * (1 + weight)` 而非标准 RMSNorm 的 `x * weight`，这之前阻止了使用 TRT-LLM 的融合 AllReduce + Residual + RMSNorm 内核。

TokenSpeed 预计算 `gemma_weight = weight + 1.0` 并将其作为 gamma 参数传递给标准融合内核——以实现 GemmaRMSNorm 通信融合。融合后，每层的 AllReduce + 残差加法 + RMSNorm 从三个独立的内核启动合并为一个：
此融合覆盖所有 Qwen3.5 解码层，并在 SM90+ 单节点 TP 部署上自动启用。

**注意力中的融合 QK-RMSNorm + 部分 RoPE + 门控分割**

在原始注意力路径中，QKV GEMM 投影后，按顺序启动 5 个独立内核来归一化、旋转和分割 Q/K/gate 向量：

| 步骤 | 操作 | 从 HBM 读取 | 写入 HBM |
|------|------|------------|---------|
| 1 | Q RMSNorm | q | q_normed |
| 2 | K RMSNorm | k | k_normed |
| 3 | Q RoPE | q_normed | q_rotated |
| 4 | K RoPE | k_normed | k_rotated |
| 5 | 门控分割 + 连续复制 | q_gate | gate |

每个中间张量（`q_normed`、`k_normed` 等）被写入全局内存，只为立即被下一个内核读取——纯粹的带宽浪费。`fused_qk_rmsnorm_rope_gate` 用单个 Triton 内核替换了所有 5 次启动。所有中间值保留在寄存器中。

**MoE 共享专家中的融合门控-Sigmoid-乘法-加法**

在 MoE 块中，共享专家输出在与路由专家输出合并之前被门控。原始代码路径为概念上的一个表达式启动 5 个独立内核：

| 步骤 | 内核 | 说明 |
|------|------|------|
| 1 | 逐元素乘法 | `h[i] * w[i]` — 每元素乘积 |
| 2 | 归约 | 对部分乘积求和 → 每 Token 1 个标量 |
| 3 | Sigmoid | `σ(gate_val)` — 对标量的逐元素操作 |
| 4 | 乘法 | `σ(gate_val) * shared_output` — 广播标量 × 完整向量 |
| 5 | 加法 | `final_hidden_states += scaled` |

关键低效之处：`gate_val` 是每 Token 的标量，但未融合路径在启动之间将每元素乘积和归约标量都具体化到 HBM。中间的 `scaled` 张量（完整的 `[num_tokens, hidden_dim]`）也被写入并立即重读。`fused_gate_sigmoid_mul_add` 在一个 Triton 内核中原地计算完整表达式 `final += σ(x·w) * shared`。点积归约、sigmoid、广播乘法和累加都在每 Token 的单个线程块内发生——中间值从不离开寄存器。

#### 3.2.3 千次同步之死

TokenSpeed 的 Decode 循环将核心前向传递——目标模型、采样器和草稿模型——捕获到单个 CUDA 图中。一旦捕获，数千个 GPU 内核以一次启动重放，完全消除每内核调度开销。

但 CUDA 图在设计上是静态的。在图重放之间，运行时仍必须在主机上执行动态工作：准备输入、解析调度索引、在投机验证后更新 Mamba 状态指针，以及协调传输状态。这些图之间的"间隙"是 CPU 开销隐藏之处——一个粗心的 .item() 或不必要的 D2H 复制可能使整个管道停顿。

TokenSpeed 将图间 CPU 开销视为一等优化目标：即使在图外，也要让主机远离关键路径。

**消除设备到主机的往返**

最隐蔽的同步模式是"无辜查询"——从 GPU 读取单个标量以在主机上做分支决策。TokenSpeed 用初始化时已知的预计算最坏情况边界来替代这些，或在 H2D 传输前捕获 CPU 端最大值，以便 GPU 张量和其边界同时可用。对于投机解码状态管理，边界检测和槽位选择使用 GPU 端哨兵值——下游内核通过边界检查跳过无效条目，而非 CPU 端过滤。整个决策树保留在设备上。

**编译融合索引运算**

混合模型中的运行时调度涉及大量索引操作：计算槽位映射、草稿 Token 布局和验证后的指针更新。在急切 PyTorch 中，每步变成一个独立的内核启动，中间值写入 HBM。TokenSpeed 使用 torch.compile 注释这些例程，允许 Inductor 将 10-14 个独立启动融合为一两个逐元素内核，所有值通过寄存器流动。GPU 保持忙碌，CPU 提交一次启动而非十四次。

**异步一切**

H2D 传输全程使用带有非阻塞复制的固定内存。传输系统轮询固定主机计数器而非调用 synchronize()，逐层加载使用基于事件的屏障，只唤醒需要数据的特定层。CPU 在当前批次仍在处理时准备下一批次。

累积效果：TokenSpeed 的 Decode 循环保持近零 CPU 开销——主机线程将时间花在提交工作上，而非等待结果。

### 3.3 FA4 支持

Flash Attention 4（FA4）是面向 NVIDIA Blackwell 架构的下一代注意力内核。Qwen3.5 默认使用 head_dim=256，对注意力计算后端提出了相当高的要求——这是一个并非所有内核都能高效开箱即用支持的配置。

对 head_dim=256 的 FA4 支持已贡献并合并到上游社区仓库。在 TokenSpeed 中，对 Qwen3.5 的原生 FA4 支持目前正在积极开发中，将在即将发布的版本中提供，进一步释放 Blackwell GPU 对 Qwen3.5 推理的全部计算潜力。

## 4. 基准测试

以 Qwen3.5-397B-A17B 为代表性示例，我们在 NVIDIA Blackwell GPU 上对 Qwen3.5 模型进行了系统性性能评估。我们要感谢 EvalScope 团队提供基准测试工具；以下报告的所有性能结果均使用 EvalScope Benchmark 获得。

**测试环境**：所有基准测试均使用 TokenSpeed 最新 Docker 镜像（lightseekorg/tokenspeed-runner:latest）进行，基于最近版本。基准测试脚本和复现说明可在 TokenSpeed 的 GitHub 仓库获取。

### 4.1 基础基准测试

我们使用固定的输入/输出长度并测量 Decode 吞吐量（输出 token/s）。我们在不同并行配置（TP/EP）下评估了不同批量大小的性能。使用了两种主要测试配置：

- 配置 1：Attn TP + MoE TP
- 配置 2：Attn TP + MoE EP

我们在 B200 上对 Qwen3.5-397B-A17B-NVFP4 的 Decode 吞吐量进行了启用和禁用 MTP 的基准测试。

在 Attn TP8 + MoE TP8 / Attn TP8 + MoE EP8 的所有输入/输出长度配置中，在 bs=1 时 MTP 提供 +100%~+159% 的吞吐量增益，此时延迟是主要瓶颈。在更高并发下，增益与输出长度密切相关：长输出工作负载（例如，输出长度 >4096）在 bs=32 / 64 时保持 +38%~+90% 的实质性加速，而短输出工作负载（例如，1024 tokens）在 bs=64 时增益减小至接近零或略微变为负值，因为当解码已经受吞吐量限制时，投机开销开始超过接受收益。

### 4.2 Agentic 工作负载基准测试

Agent 应用的快速扩散——涵盖工具调用历史和多轮对话上下文——从根本上重塑了生产工作负载的特征。为了反映真实世界的 Agent 行为，我们使用 Agentic 工作负载测试套件，模拟真实的 Agent 调用模式（50K 第一轮上下文，每轮追加 800 tokens，共 10-15 轮）。

在 B200 上使用 NVFP4，TokenSpeed 在 Agentic 工作负载下为 Qwen3.5-397B-A17B 提供卓越的单用户吞吐量。所有四种并行配置——TP4、TP4EP4、TP8 和 TP8EP8——在 bs=1 时保持 500+ tok/s，TP8 实现约 580 tok/s 的峰值。

在 concurrent=16 时，TP4 系列扩展到约 2K tok/min/GPU 系统吞吐量，而 TP8 系列达到约 1K tok/min/GPU。相同 GPU 数量下的纯 TP 和 TP+EP 配置表现出相当的吞吐量-延迟权衡，在不牺牲性能的情况下给用户提供部署灵活性。值得注意的是，多轮 Agentic 工作负载实现了超过 90% 的平均 KV 缓存命中率，显著减少了 Prefill 开销并贡献于整体吞吐量增益。

### 4.3 高达 1M 的长上下文基准测试

长上下文处理是 Agent 工作负载中的另一个关键挑战。虽然前缀缓存可以命中多轮对话中大量重复的前缀并显著减少 Prefill 开销，但 Decode 阶段在每一步仍必须读取并关注完整的历史 KV，这是缓存无法绕过的——上下文越长，每步 Decode 的内存访问成本就越高。

基于 NIAH（Needle-in-a-Haystack）1M 样本，我们切分了四种提示长度——128K / 256K / 512K / 1M——进行评估。在 Qwen3.5-397B-A17B 上，Decode 吞吐量在 128K 内保持约 530 tok/s/用户，256K 约 495，1M 约 445（在 TP8 上测量），从 128K 到 1M 的端到端退化仅约 16%——长上下文吞吐量衰减得到了良好控制。

## 5. 结论

通过上述优化和架构设计，TokenSpeed 为 Qwen3.5 模型提供了出色的性能——特别是在 Agentic 工作负载中，实现了超低延迟生成和高推理吞吐量。TokenSpeed 将继续突破 Qwen 推理优化的边界，在堆栈的每个层次追求更极限的性能。
